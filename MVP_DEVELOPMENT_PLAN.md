# ReadBridge MVP 开发任务清单与版本路线图

## 1. 当前阶段判断

ReadBridge 目前已经从产品想法推进到可交互原型阶段。

当前原型已经覆盖主要学习闭环：

选择文章 / 导入 PDF -> 阅读 -> 收藏生词 -> 写 summary -> AI 辅助修改 -> 收藏短语 / 句型 -> 生词复习 -> 音频听读 -> 原句生成记忆画面

下一阶段的目标不是继续堆功能，而是把原型收敛成一个稳定、可测试、可迭代的 MVP。

## 2. MVP 核心闭环

第一版产品必须优先保证下面这条路径顺畅：

1. 用户选择平台文章或导入自己的 PDF。
2. 用户进入阅读页并开始阅读。
3. 用户遇到生词后收藏。
4. 系统保存生词、释义、原句和来源文章。
5. 用户读后写一个 summary。
6. 用户使用 AI 修改 summary。
7. 用户把修改稿中的短语或句型加入生词本。
8. 用户回到生词本复习。
9. 用户通过音频朗读和记忆画面辅助理解与记忆。

如果后续开发资源有限，应优先保证这条路径稳定。

## 3. 功能优先级

### P0：必须完成

- 文章列表与文章阅读页
- 生词收藏
- 生词本复习
- Summary 写作与保存
- 阅读记录
- 本地数据持久化
- 基础响应式布局

### P1：MVP 增强体验

- PDF 导入
- AI 辅助修改 summary
- 从 AI 修改稿收藏短语 / 句型
- 浏览器音频朗读
- 生词原句生成记忆画面
- 按文章筛选生词

### P2：后续迭代

- 用户账号
- 云端同步
- 更完整的 PDF.js 解析
- 扫描版 PDF OCR
- 真实 AI summary 修改接口
- 真实 AI 图片生成接口
- 更自然的 TTS 语音
- 间隔重复复习算法
- 用户自定义词义

## 4. 开发任务拆分

### 4.1 前端结构

- 建立正式前端项目结构。
- 将当前单文件原型拆成页面和组件。
- 建议拆分：
  - ArticleLibrary
  - PdfImporter
  - Reader
  - AudioControls
  - SummaryEditor
  - AiRevisionPanel
  - VocabularyPanel
  - VocabularyReview
  - MemoryImageCard
  - ReadingRecords

### 4.2 数据模型

需要明确并稳定以下数据结构：

- Article
- ImportedArticle
- VocabularyItem
- ReadingSummary
- ReadingRecord
- MemoryImage

需要重点处理：

- 数据版本迁移
- 本地存储异常
- 重复收藏
- 删除文章后关联数据如何处理
- 导入 PDF 后的文章 ID 稳定性

### 4.3 阅读页

任务：

- 支持文章正文展示。
- 支持段落选择。
- 支持点击单词收藏。
- 支持当前文章生词列表。
- 支持标记已读。
- 支持音频朗读。

验收：

- 用户可以从文章列表进入阅读页。
- 点击英文单词可以收藏。
- 收藏后本篇生词区立即更新。
- 切换文章时不会残留上一文章的朗读或 summary 状态。

### 4.4 PDF 导入

任务：

- 拖拽导入 PDF。
- 点击选择 PDF。
- 提取 PDF 文本。
- 生成自定义文章。
- 导入失败时给出清晰提示。

建议：

- MVP 可继续使用本地解析。
- V1 应引入 PDF.js。
- 扫描版 PDF 暂不作为 MVP 必做。

验收：

- 文本型 PDF 可以进入文章列表。
- 导入文章可以正常阅读、收藏生词、写 summary。
- 扫描版或解析失败时不崩溃。

### 4.5 生词本

任务：

- 展示所有收藏项。
- 区分单词、短语、句型。
- 展示释义、原句、来源文章。
- 支持标记已复习。
- 支持按来源文章筛选。
- 支持生成记忆画面。

验收：

- 用户能看懂每个词来自哪里。
- 用户能通过原句回忆语境。
- 生成的记忆画面和生词关联保存。

### 4.6 Summary 与 AI 修改

任务：

- 保存用户 summary。
- 展示 AI 修改建议。
- 支持使用修改稿替换用户文本。
- 支持从修改稿收藏短语 / 句型。

建议：

- 当前原型为本地模拟。
- V1 可接入真实 AI API。
- 接入真实 API 前，先设计 prompt 和返回数据结构。

验收：

- 用户可以保存 summary。
- 用户可以看到修改稿。
- 用户可以把短语 / 句型加入生词本。
- AI 失败时不影响 summary 保存。

### 4.7 音频朗读

任务：

- 支持整篇文章朗读。
- 支持当前段落朗读。
- 支持语速调节。
- 支持声音选择。
- 支持播放、暂停、停止。

验收：

- 切换文章时停止旧朗读。
- 切换段落模式后，用户能明确当前朗读段落。
- 不支持朗读的浏览器给出提示。

### 4.8 记忆画面

任务：

- 从生词所在原句生成记忆画面。
- 展示画面。
- 支持重新生成。
- 支持下载图片。

建议：

- MVP 使用本地图像卡片生成。
- 后续可接入 AI 图片生成。
- 图像生成应保留 prompt，方便用户理解画面来源。

验收：

- 每个生词都可以生成画面。
- 图片刷新后仍然保留。
- 移动端显示不溢出。

## 5. 技术路线建议

### 5.1 快速 MVP

适合继续快速验证：

- Vite
- React
- localStorage / IndexedDB
- PDF.js
- Web Speech API
- 本地图像卡片生成

优点：

- 开发快。
- 部署简单。
- 不需要后端。
- 适合早期用户测试。

### 5.2 V1 产品化

适合准备正式使用：

- Next.js 或 React + Vite
- Supabase / Firebase / 自建后端
- 数据库保存文章、生词、summary、记忆图
- OpenAI API 用于 summary 修改
- 图像生成服务用于记忆画面
- TTS 服务用于更自然音频

优点：

- 支持账号和同步。
- 数据更安全。
- AI 能力更真实。
- 更适合长期使用。

## 6. 用户测试计划

### 6.1 测试对象

建议先找 3-5 个英语兴趣学习者。

用户不需要是考试人群，最好是：

- 喜欢英文阅读
- 读英文时会查词
- 愿意读后写一点东西
- 对阅读材料有自己的兴趣

### 6.2 测试任务

让每个用户完成以下任务：

1. 选择一篇文章开始阅读。
2. 收藏 2-3 个生词。
3. 写一个 3 句话 summary。
4. 使用 AI 辅助修改 summary。
5. 收藏一个修改稿短语。
6. 回到生词本复习。
7. 为一个生词生成记忆画面。
8. 尝试播放文章音频。
9. 尝试导入自己的 PDF。

### 6.3 观察问题

重点观察：

- 用户能否自然理解产品要做什么。
- 用户是否知道下一步该点哪里。
- 生词收藏是否打断阅读。
- Summary 写作是否压力太大。
- AI 修改是否真的有帮助。
- 记忆画面是否帮助记忆，还是只觉得新奇。
- PDF 导入是否是用户真正需要的入口。
- 用户是否愿意第二天回来复习。

### 6.4 访谈问题

测试后可以问：

- 你最想继续使用哪个功能？
- 哪个功能你觉得可有可无？
- 你阅读时最卡的地方是什么？
- 生词本对你有帮助吗？
- Summary 修改对你有帮助吗？
- 你会导入自己的书或文章吗？
- 如果明天继续用，你会先打开哪里？

## 7. 版本路线图

### v0.1：交互原型

目标：

- 跑通完整体验。
- 帮助确认产品方向。

状态：

- 已完成。

### v0.2：可测试 MVP

目标：

- 将静态原型升级为结构化前端项目。
- 修复明显交互问题。
- 支持本地数据稳定保存。
- 准备给真实用户试用。

建议开发重点：

- React / Vite 项目化
- 组件拆分
- IndexedDB 替代单纯 localStorage
- PDF.js 接入
- 基础错误处理
- 用户测试记录表

### v0.3：AI 能力接入

目标：

- 把本地模拟 AI 改成真实 AI 服务。

建议开发重点：

- Summary 修改 API
- 短语 / 句型提取 API
- 生词释义 API
- 记忆画面生成 API
- AI 失败兜底

### v0.4：复习体验强化

目标：

- 让用户愿意回来复习。

建议开发重点：

- 今日复习
- 已掌握 / 仍模糊状态
- 按文章回顾
- 间隔重复
- Summary 回顾

### v1.0：正式个人学习工具

目标：

- 可以作为长期使用的英语阅读工具。

建议开发重点：

- 用户账号
- 云端同步
- 数据备份
- 更好的 PDF 解析
- 更自然音频
- AI 图像与 AI 写作稳定接入
- 部署上线

## 8. 当前不建议继续做的事

短期内不建议继续新增：

- 社区功能
- 排行榜
- 复杂积分系统
- 课程体系
- 大量考试题库
- 多人协作
- 支付订阅

这些功能会把产品推向另一个方向，容易稀释当前最有价值的阅读闭环。

## 9. 下一步行动建议

最推荐的下一步：

1. 将当前静态原型迁移为 React / Vite 项目。
2. 保持功能不扩张，只做结构化和稳定性。
3. 准备 3-5 人用户测试。
4. 根据测试结果决定 v0.3 先接 AI 修改，还是先强化 PDF 导入。
